# Phase 4: 改善提案

## 目的

問題のある結合に対して、具体的な改善策を提案する。

## 改善パターン

詳細: [PATTERNS.md](PATTERNS.md)

### クイックリファレンス

| 問題 | 条件 | 解決策 |
|------|------|--------|
| S高すぎ | S > D | インターフェース, DTO, イベント駆動 |
| D遠すぎ | D > S+3 | コロケーション, モジュール統合 |
| D近すぎ | D < S-3 | モジュール分離, BC定義 |
| V高依存 | V高 & S高 | ACL, 公開ホストサービス |

## 優先度判定

| 優先度 | 条件 | 理由 |
|--------|------|------|
| 🔴 高 | B≤3 & V≥8 | 変更のたびに問題発生 |
| ⚠️ 中 | B=4-6 または 下流依存多 | 影響範囲が大きい |
| 💡 低 | B=5-6 & V≤4 | 安定しているため急がない |

## 出力

```markdown
## Phase 4: 改善提案

### 🔴 高優先度

#### CoreService への依存

**現状**: S=7, D=6, V=9, B=2
**問題**: S≈Dで均衡度が低い（分散モノリス傾向）、かつ高変動性

**改善案**: インターフェース導入 + 距離の調整

```php
// Before
class OrderService {
    public function __construct(
        private CorePricingService $pricing  // 具象依存、同一BC内
    ) {}
}

// After
class OrderService {
    public function __construct(
        private PricingServiceInterface $pricing  // 抽象依存
    ) {}
}
// CorePricingServiceをpackages/pricing/に分離
```

**期待効果**: S: 7→3, D: 6→8, B: 2→8
**工数**: 中
**トレードオフ**: インターフェース追加による間接化
