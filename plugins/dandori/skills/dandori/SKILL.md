# 段取り八分 (dandori) — 着手前に前提を固める検討プロトコル

## なぜこれが要るか

長文の意思決定資料・提言書・設計書・比較資料を作るとき、最大の手戻り要因は
**前提条件が中盤〜終盤で「後出し」される**ことだ。固定前提（既存システムの稼働状態）、
着手時期、工数の測り方、対象環境、意思決定の優先軸——こうした前提が途中で判明するたびに、
案A vs B の比較・工数見積もり・章立てといった「上物」を丸ごと作り直すことになる。

これを防ぐ唯一の手は、**上物を組む前に前提を地面として固めること**。
着手の最初のターンでこのプロトコルを回す。

## persistence ルールとの関係（重要）

ユーザーの基本方針は「人間に前提確認を求めず、最も合理的な仮定で進め、仮定は記録せよ」。
このスキルはそれと**衝突しない**。やることはこうだ:

- **逆質問で止まらない。** Claude 自身が最も合理的な仮定で前提シートを**埋める**。
- 埋めたシートを**1回だけ**提示し、「この前提で進めます。違う点だけ直してください」と渡す。
- 返答がなければ・「進めて」なら、その仮定のまま着手する（＝記録された仮定で前進）。

つまり「開いた逆質問のラリー」ではなく「仮定の一括提示 → ズレだけ訂正 → 前進」。
これは persistence の精神（仮定して進め、記録せよ）を満たしたうえで、前提ズレによる手戻りを消す。

## Step 1 — 前提シートを埋めて提示する

着手の最初の応答で、依頼内容・コードベース・過去の文脈・メモリから読み取れる範囲で
**最も合理的な仮定**を立て、以下5項目を埋める。空欄で聞き返すのではなく、必ず仮定で埋めること。
仮定の確度が低い項目には `（仮定・要確認）` を付けて、ユーザーの目を誘導する。

ALWAYS use this exact template:

```
## 前提シート（私の仮定です。違う項目だけ直してください。なければそのまま進めます）

**固定前提（変更しない事実）**
- 既存システムの稼働状態 / 対象環境 / 運用・技術スタック前提（AIペアプロ・既存ライブラリ活用の有無 等）

**時期前提**
- 着手見込み時期 / リードタイム / 外部依存（他チーム合意・締切・監査時期 等）

**意思決定軸（優先順）**
1. （例）早期実効化  2. 工数最小化  3. 監査説明性  ← この順序が比較や推奨の軸になる

**成果物の形**
- 媒体（HTML / MD / Notion 等） / 想定読者と技術リテラシ / 目標分量 / 粒度（戦略レベル or 実装詳細レベル）

**スコープ境界**
- 何を解決し、何を解決しないか（他プロジェクトや別フェーズに委ねる範囲）

→ この前提で進めます。修正点はありますか？
```

意思決定軸が2〜3択で迷う・成果物の媒体や粒度が複数あり得るなど、
**少数の二者択一で大勢が決まる**場合は AskUserQuestion を1回使ってまとめて確定してもよい。
ただし自由記述で埋まる項目まで質問に分解しないこと（ラリーを増やすだけ）。

## Step 2 — 確定した前提を「ロック」する

ユーザーの訂正を反映したら、確定事項を明示的に固定する。
着手後に「これは前回決めたはず」が緩んで再議論になるのを防ぐため、
成果物の冒頭か作業メモに「決定済み前提」として残し、以後はそれを参照する。

固定前提・意思決定軸・スコープ境界の3つは特に再燃しやすい。
これらに触れる変更が出たら「これは決定済み前提XXに反します。前提ごと見直しますか？」と確認する。

## Step 3 — 構造変更は「提案→承認→一括実装」で回す

着手後、大きな構造変更が必要になったとき、**1箇所ずつの修正指示を連鎖させない**。
「3章タイトル変更」「4.3削除」「章10削除」を別々の往復でやると、整合性が崩れて結局やり直しになる。

代わりに:

1. **提案する** — 「意思決定資料として再編する観点で、全体の見直し案を出します」と、変更のまとまりを一括提示。
2. **影響範囲を先回りする** — 「案B単独にすると章8・章10が不要になります。一緒に消しますか？」のように、
   一つの変更が波及する先を**変更前に**示す。ユーザーが部分指示しか出していなくても、整合に必要な連鎖を提案に含める。
3. **承認後に一括反映** — OK が出た範囲をまとめて実装する。

## Step 4 — 「決定」と「思考過程」を分ける

「議論 → 実装 → 見て『あれ違う』→ 議論し直し → 実装し直し」のループは高くつく。
**議論 → 結論を文書化 → 確認 → 一括実装** の順にする。
実装前に結論を確認するため、plan モードの活用が有効なら提案する。
